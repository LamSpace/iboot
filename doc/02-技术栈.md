## 引言

iBoot 项目采用现代化的前后端分离架构，技术选型经过精心挑选，旨在提供一个高效、安全、可扩展的企业级管理系统解决方案。本章详细介绍项目所采用的各项技术及其选型理由。

**Spring Security 功能：**

- **认证方式**：表单认证、JWT 认证、OAuth2 认证

**JWT 认证详解：**

JWT（JSON Web Token）是一种无状态认证机制，由三部分组成：
- **Header**：令牌类型和加密算法
- **Payload**：用户信息和声明
- **Signature**：签名验证

**JWT 认证流程：**
```
1. 用户登录，提交用户名和密码
2. 服务端验证凭证，生成 JWT Token
   - 使用私钥对 Header 和 Payload 签名
   - 设置过期时间（expiration）
3. 客户端存储 Token（localStorage/SessionStorage）
4. 后续请求在 Authorization Header 中携带 Token
   - 格式：Bearer <token>
5. 服务端验证 Token
   - 验证签名有效性
   - 检查是否过期
   - 解析用户信息到 SecurityContext
6. Token 过期处理
   - 滑动过期机制（Redis 控制会话）
   - 刷新 Token 机制（可选）
```

**JWT 配置示例：**
```yaml
jwt:
  secret: your-secret-key-minimum-256-bits-for-hs256
  expiration: 604800        # 7 天（秒）
  session-timeout: 30       # 30 分钟滑动过期
  token-prefix: Bearer
  header: Authorization
```

**密码加密：**
```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // 12 轮加密
}
```

### 2.13 并发与线程

| 技术 | 版本 | 描述 |
|------|------|------|
| Virtual Threads | JDK 25 (JEP 444) | 轻量级虚拟线程，高并发场景性能优异 |
| Platform Threads | JDK 25 | 传统 OS 线程，一对一映射 |

**虚拟线程架构：**

```
┌─────────────────────────────────────────────┐
│           应用层（Virtual Threads）          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│  │ VT #1   │  │ VT #2   │  │ VT #N   │    │
│  └────┬────┘  └────┬────┘  └────┬────┘    │
│       │           │           │           │
│       ▼           ▼           ▼           │
│  ┌─────────────────────────────────────┐  │
│  │   Carrier Thread (Platform)         │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

**虚拟线程特性：**
- **轻量级**：栈空间仅几 KB，可创建百万级并发
- **弹性伸缩**：按需创建，无需线程池
- **I/O 自动卸载**：阻塞时自动让出 CPU
- **兼容现有 API**：支持 `Thread`、`ExecutorService` 等标准 API

**配置示例：**
```yaml
# 启用 Spring 虚拟线程
spring:
  threads:
    virtual:
      enabled: true

# 启用 Tomcat 虚拟线程
server:
  tomcat:
    threads:
      virtual:
        enabled: true
    max-connections: 10000
```

**异步任务执行器：**
```java
@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        return Executors.newVirtualThreadPerTaskExecutor();
    }
}
```

**性能对比：**

| 指标 | Platform Threads | Virtual Threads |
|------|------------------|-----------------|
| 线程创建开销 | ~1MB 栈空间 | ~几 KB 栈空间 |
| 最大并发数 | 数千 | 百万级 |
| 上下文切换 | OS 级，开销大 | 用户态，开销小 |
| I/O 阻塞 | 占用 OS 线程 | 自动卸载，不占用 |
| 适用场景 | CPU 密集型 | I/O 密集型 |

**注意事项：**
- 不适用于 CPU 密集型计算（建议使用并行流）
- `synchronized` 会阻塞虚拟线程（优先使用 `ReentrantLock`）
- `ThreadLocal` 需显式清理（已正确实现）

**详细文档：** [16-虚拟线程指南](./16-虚拟线程指南.md)
```
1. 客户端请求上传 URL
2. 服务端生成预签名 PUT URL
3. 客户端直接上传到 MinIO
4. 上传完成回调服务端
5. 服务端保存文件元数据到数据库
```

### 2.15 消息推送

- **组播（Multicast）**：向指定用户组推送消息
- **广播（Broadcast）**：向所有在线用户推送消息

**集群支持：**
- 单机模式：使用本地 SseEmitterManager 管理连接
- 集群模式：通过 Redis Pub/Sub 实现跨实例消息广播

**MapStruct 对象映射：**

MapStruct 是一个高性能的对象映射框架，在编译时生成类型安全的映射代码。

**核心特性：**
- **编译时代码生成**：运行时无开销，性能优秀

- **Lombok 集成**：通过 lombok-mapstruct-binding 与 Lombok 无缝协作

**使用示例：**
```java
@Mapper(componentModel = "spring", injectionStrategy = InjectionStrategy.CONSTRUCTOR)
public interface UserMapper {

    // DTO 转 Entity
    @Mapping(target = "id", ignore = true)
    @Mapping(target = "avatar", defaultValue = "")
    User toEntity(CreateUserRequest request);

    // Entity 转 DTO
    @Mapping(target = "roleNames", source = "roles")
    UserResponse toResponse(User user);

    // List 映射（自动遍历）
    List<UserResponse> toResponseList(List<User> users);
}
```

**常用注解：**

| 注解 | 描述 | 示例 |
|------|------|------|
| `@Mapping(source, target)` | 字段映射 | `@Mapping(source = "userName", target = "username")` |
| `@Mapping(ignore = true)` | 忽略字段 | `@Mapping(target = "id", ignore = true)` |
| `@Mapping(defaultValue)` | 默认值 | `@Mapping(target = "status", defaultValue = "1")` |
| `@Mappings` | 多个映射 | `@Mappings({@Mapping(...), @Mapping(...)})` |
| `@Named` | 自定义映射方法 | `@Mapping(qualifiedByName = "formatDate")` |

**Mapper 使用方式：**
```java
@RestController
@RequestMapping("/users")
public class UserController {
    private final UserMapper userMapper;

    // 构造器注入
    public UserController(UserMapper userMapper) {
        this.userMapper = userMapper;
    }

    public List<UserResponse> list() {
        List<User> users = userService.findAll();
        // MapStruct 自动处理 List 遍历
        return userMapper.toResponseList(users);
    }
}
```

**生成代码位置：**
编译后生成的实现代码位于 `target/generated-sources/annotations/` 目录下。

**注意事项：**
1. 映射接口必须使用 `interface`，不能使用抽象类
2. 字段名相同时可自动映射，无需 `@Mapping` 注解
3. 复杂对象映射建议使用 `source` 属性明确指定源字段
4. 与 Lombok 合用时，确保 `lombok-mapstruct-binding` 依赖已添加

**Spring Boot 选择理由：**
- **约定优于配置**：大量自动配置，减少 XML 和样板代码
- **内嵌容器**：无需部署 WAR，直接运行 JAR
- **生产就绪**：Actuator 提供健康检查、指标监控
- **微服务友好**：与 Spring Cloud 无缝集成
- **生态丰富**：几乎所有 Java 库都有 Spring Boot Starter

**MyBatis vs JPA：**
| 对比项 | MyBatis | JPA/Hibernate |
|--------|---------|---------------|
| SQL 控制 | 完全控制，手写 SQL | 自动生成，HQL/ Criteria |
| 学习曲线 | 低，会 SQL 即可 | 较高，需要理解 ORM 概念 |
| 复杂查询 | 灵活，支持复杂 SQL | 较复杂，JPQL 有限制 |
| 性能优化 | 容易优化 | 需要理解 N+1 等问题 |
| 适用场景 | 复杂业务、SQL 优化要求高 | 简单 CRUD、领域模型清晰 |

**Druid vs HikariCP：**
| 对比项 | Druid | HikariCP |
|--------|-------|----------|
| 性能 | 优秀 | 极快（Spring Boot 默认） |
| 监控 | 内置强大监控 | 简单 |
| 防火墙 | WallFilter 防 SQL 注入 | 无 |
| 扩展性 | 插件丰富 | 简洁 |
| 选择建议 | 需要监控和 SQL 分析 | 追求极致性能 |

**SSE vs WebSocket：**

| 对比项 | SSE | WebSocket |
|--------|-----|-----------|
| 通信方向 | 单向（服务器→客户端） | 双向 |
| 协议 | HTTP | WebSocket（ws://） |
| 浏览器支持 | 原生支持（EventSource） | 原生支持（WebSocket） |
| 连接建立 | 简单（HTTP GET） | 需要握手协议 |
| 消息格式 | text/event-stream | 二进制/文本 |
| 自动重连 | 浏览器自动处理 | 需要手动实现 |
| 服务端复杂度 | 低 | 较高 |
| 适用场景 | 消息推送、通知、实时更新 | 聊天室、在线游戏、协同编辑 |
| 选择建议 | 单向推送场景优先选择 SSE | 双向交互场景选择 WebSocket |

SSE 是消息推送场景的理想选择，特别是在管理系统中用于通知、消息提醒等场景。相比 WebSocket，SSE 具有以下优势：
- **开发简单**：服务端使用 SseEmitter 即可实现，客户端使用 EventSource API
- **协议简单**：基于标准 HTTP，无需额外协议支持
- **自动重连**：浏览器自动处理断线重连，无需手动实现
- **轻量级**：消息格式简单，开销更小
- **防火墙友好**：使用标准 HTTP 端口，不易被拦截

详细实现和使用指南请参考 [15-SSE 推送集成](./15-SSE 推送集成.md)。
