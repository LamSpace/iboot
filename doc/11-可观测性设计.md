## 概述

### 11.1 可观测性三大支柱

iBoot 项目基于现代化的可观测性技术栈，提供完整的系统观测能力：

| 支柱 | 技术栈 | 用途 | 数据保留 |
|------|--------|------|----------|
| **Metrics（指标）** | Prometheus + Thanos + Grafana | 数值型监控数据，支持聚合计算和趋势分析 | 原始数据 30 天，5 分钟聚合 90 天，1 小时聚合 1 年 |
| **Logging（日志）** | ELK Stack（Elasticsearch + Logstash + Kibana） | 结构化日志收集、存储和检索 | 默认 30 天，可配置 |
| **Tracing（追踪）** | Micrometer Tracing（可选） | 分布式链路追踪，分析请求链路 | 随日志存储周期 |

### 11.5 数据流说明

**指标数据流（Metrics Pipeline）：**

1. **采集层**
   - iboot-admin 应用通过 Micrometer 暴露 `/actuator/prometheus` 端点
   - Exporters 采集基础设施指标（MySQL、Redis、Node、Elasticsearch）
   - 采集频率：应用指标 10s，基础设施指标 15s

2. **传输层**
   - Prometheus 通过 Pull 模式定期抓取目标端点
   - 支持静态配置、文件服务发现、DNS 服务发现

3. **存储层**
   - Prometheus TSDB 存储近期数据（本地 24h）
   - Thanos Sidecar 实时上传数据块到 MinIO 对象存储
   - Thanos Store 从对象存储加载历史数据
   - Thanos Compact 执行数据压缩和降分辨率（30d 原始 → 90d 5m → 1y 1h）

4. **查询层**
   - Thanos Query 提供统一查询入口
   - 支持 PromQL 查询语言
   - 自动合并实时数据和历史数据

5. **展示层**
   - Grafana 连接 Thanos Query 作为数据源
   - 预配置监控仪表板
   - 支持自定义 PromQL 查询

**日志数据流（Logging Pipeline）：**

1. **采集层**
   - 应用通过 Logback 输出日志到文件
   - 同时通过 TCP 发送结构化 JSON 日志到 Logstash

2. **传输层**
   - Logstash 监听 5044 端口接收 Beats 协议
   - 监听 5000 端口接收 TCP JSON 日志

3. **存储层**
   - Logstash 处理后写入 Elasticsearch
   - 按日期自动创建索引（iboot-admin-YYYY.MM.dd）

4. **展示层**
   - Kibana 连接 Elasticsearch
   - 支持全文搜索、字段过滤、可视化图表

### 11.7 自定义业务指标

**BusinessMetricsService 实现：**

```java
@Service
@Slf4j
public class BusinessMetricsService {

    @Autowired
    private MeterRegistry meterRegistry;

    // 用户登录指标
    private Counter userLoginSuccessCounter;
    private Counter userLoginFailedCounter;
    private AtomicInteger onlineUserCount;

    // 业务流程指标
    private Timer businessProcessTimer;
    private Counter businessProcessSuccessCounter;
    private Counter businessProcessFailedCounter;

    // 缓存指标
    private Gauge cacheHitRateGauge;
    private AtomicInteger cacheHitCount;
    private AtomicInteger cacheTotalCount;

    // 文件操作指标
    private Counter fileUploadCounter;
    private Counter fileDownloadCounter;
    private DistributionSummary fileUploadSizeSummary;

    @PostConstruct
    public void init() {
        // 登录成功次数
        userLoginSuccessCounter = Counter.builder("user_login_success")
                .description("用户登录成功次数")
                .tag("application", "iboot-admin")
                .register(meterRegistry);

        // 登录失败次数
        userLoginFailedCounter = Counter.builder("user_login_failed")
                .description("用户登录失败次数")
                .tag("application", "iboot-admin")
                .register(meterRegistry);

        // 在线用户数（Gauge 实时值）
        onlineUserCount = new AtomicInteger(0);
        Gauge.builder("user_online_count", onlineUserCount, AtomicInteger::get)
                .description("在线用户数量")
                .tag("application", "iboot-admin")
                .register(meterRegistry);

        // 业务流程处理耗时（Timer）
        businessProcessTimer = Timer.builder("business_process_duration")
                .description("业务流程处理耗时")
                .tag("application", "iboot-admin")
                .register(meterRegistry);

        // 缓存命中率（Gauge 百分比）
        cacheHitRateGauge = Gauge.builder("cache_hit_rate_percent", this, this::calculateCacheHitRate)
                .description("缓存命中率")
                .tag("application", "iboot-admin")
                .baseUnit(BaseUnits.PERCENT)
                .register(meterRegistry);
    }

    // 记录登录成功
    public void recordUserLoginSuccess() {
        userLoginSuccessCounter.increment();
        onlineUserCount.incrementAndGet();
    }

    // 记录登录失败
    public void recordUserLoginFailed() {
        userLoginFailedCounter.increment();
    }

    // 记录缓存命中
    public void recordCacheHit() {
        cacheHitCount.incrementAndGet();
        cacheTotalCount.incrementAndGet();
    }

    // 记录缓存未命中
    public void recordCacheMiss() {
        cacheTotalCount.incrementAndGet();
    }

    private double calculateCacheHitRate() {
        int total = cacheTotalCount.get();
        if (total == 0) return 0.0;
        return (double) cacheHitCount.get() / total * 100;
    }
}
```

**使用示例：**

```java
@Service
public class AuthService {

    @Autowired
    private BusinessMetricsService metricsService;

    public LoginResult login(String username, String password) {
        try {
            // 认证逻辑
            User user = authenticate(username, password);

            // 记录登录成功
            metricsService.recordUserLoginSuccess();

            return LoginResult.success(user);

        } catch (AuthenticationException e) {
            // 记录登录失败
            metricsService.recordUserLoginFailed();
            throw e;
        }
    }

    public <T> T executeBusinessProcess(String processType, Supplier<T> process) {
        // 开始计时
        Timer.Sample sample = metricsService.startBusinessProcess();

        try {
            T result = process.get();

            // 记录成功
            metricsService.recordBusinessProcessSuccess(sample, processType);

            return result;

        } catch (Exception e) {
            // 记录失败
            metricsService.recordBusinessProcessFailed(sample, processType);
            throw e;
        }
    }
}
```

### 11.9 Druid 连接池监控

**DruidMetricsBinder 实现：**

```java
@Component
public class DruidMetricsBinder {

    @Autowired
    private DataSource dataSource;

    @Autowired
    private MeterRegistry meterRegistry;

    @PostConstruct
    public void bindMetrics() {
        if (dataSource instanceof DruidDataSource) {
            DruidDataSource druidDataSource = (DruidDataSource) dataSource;

            // 活跃连接数
            Gauge.builder("db.connection.active", druidDataSource, DruidDataSource::getActiveCount)
                    .description("数据库活跃连接数")
                    .tag("pool", "druid")
                    .register(meterRegistry);

            // 空闲连接数
            Gauge.builder("db.connection.idle", druidDataSource, DruidDataSource::getPoolConnectionCount)
                    .description("数据库空闲连接数")
                    .tag("pool", "druid")
                    .register(meterRegistry);

            // 等待连接数
            Gauge.builder("db.connection.wait", druidDataSource, DruidDataSource::getWaitThreadCount)
                    .description("等待获取连接的线程数")
                    .tag("pool", "druid")
                    .register(meterRegistry);
        }
    }
}
```

### 11.10 HTTP 请求监控过滤器

**MonitorFilter 实现：**

```java
@Component
@Order(1)
public class MonitorFilter implements Filter {

    @Autowired
    private MonitorUtils monitorUtils;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;

        String method = httpRequest.getMethod();
        String uri = httpRequest.getRequestURI();

        long startTime = System.currentTimeMillis();

        try {
            chain.doFilter(request, response);
        } finally {
            long duration = System.currentTimeMillis() - startTime;
            String outcome = String.valueOf(httpResponse.getStatus()).startsWith("2") ? "SUCCESS" : "FAILURE";

            // 记录请求计数和耗时
            monitorUtils.incrementRequest(uri, method, outcome);

            Timer.Sample sample = Timer.start(monitorUtils.getMeterRegistry());
            sample.stop(Timer.builder("http.server.requests")
                    .tag("method", method)
                    .tag("uri", uri)
                    .tag("status", String.valueOf(httpResponse.getStatus()))
                    .tag("outcome", outcome)
                    .register(monitorUtils.getMeterRegistry()));
        }
    }
}
```

### 11.13 Logstash 配置

**logstash.conf：**

```conf
input {
    # TCP 输入（Beats 协议）
    beats {
        port => 5044
        host => "0.0.0.0"
    }

    # TCP 输入（JSON 格式）
    tcp {
        port => 5000
        host => "0.0.0.0"
        codec => json
    }
}

filter {
    # 添加处理时间戳
    mutate {
        add_field => { "processed_at" => "%{@timestamp}" }
    }

    # 解析应用日志
    if [app] == "iboot-admin" {
        # 提取日志级别
        mutate {
            lowercase => [ "level" ]
        }

        # 提取错误堆栈
        if [stack_trace] {
            mutate {
                gsub => [ "stack_trace", "\n", "\\n" ]
            }
        }
    }

    # GeoIP 解析（如有 IP 字段）
    if [client_ip] {
        geoip {
            source => "client_ip"
            target => "geoip"
        }
    }
}

output {
    # 输出到 Elasticsearch
    elasticsearch {
        hosts => ["elasticsearch:9200"]
        index => "iboot-admin-%{+YYYY.MM.dd}"
        template_name => "iboot-admin"
        template_overwrite => true
    }

    # 调试输出（可选）
    stdout {
        codec => rubydebug
    }
}
```

### 11.22 数据保留策略

| 分辨率 | 保留时间 | 说明 |
|--------|----------|------|
| 原始数据（Raw） | 30 天 | 15s 采样间隔的原始指标数据 |
| 5 分钟聚合 | 90 天 | 每 5 分钟聚合一次，存储空间减少约 20 倍 |
| 1 小时聚合 | 1 年 | 每小时聚合一次，存储空间减少约 240 倍 |

**Compact 组件自动执行降分辨率：**

## 监控指标详解

### 11.23 指标类型说明

| 类型 | 描述 | 使用场景 | 示例 |
|------|------|----------|------|
| Counter | 只增计数器，重启归零 | 请求次数、错误次数 | `user_login_success_total` |
| Gauge | 可增可减的瞬时值 | 内存使用、在线用户数 | `user_online_count` |
| Histogram | 直方图，支持分位数计算 | 请求延迟、响应大小 | `http_server_requests_seconds` |
| Summary | 服务端计算分位数 | 对分位数精度要求高的场景 | - |
| Timer | Micrometer 封装，基于 Histogram | 方法耗时统计 | `@Timed` 注解 |

### 11.24 核心业务指标清单

| 指标名称 | 类型 | 描述 | 标签 |
|----------|------|------|------|
| `user_login_success_total` | Counter | 用户登录成功次数 | application, namespace |
| `user_login_failed_total` | Counter | 用户登录失败次数 | application, namespace |
| `user_online_count` | Gauge | 当前在线用户数量 | application, namespace |

**SSE 推送指标说明：**

- `push_connection_active`：实时反映当前建立 SSE 连接的用户数量，可用于评估系统负载
- `push_event_sent_total`：累计推送成功的事件数量，可用于计算推送速率
- `push_event_failed_total`：累计推送失败的事件数量，结合成功次数可计算推送成功率

**推送成功率计算：**
```promql
# 推送成功率（5 分钟滑动窗口）
rate(push_event_sent_total[5m]) / (rate(push_event_sent_total[5m]) + rate(push_event_failed_total[5m])) * 100
```

```java
@Component
public class BusinessMetricsService {

    private AtomicInteger pushConnectionCount;
    private Counter pushEventSentCounter;
    private Counter pushEventFailedCounter;

    @PostConstruct
    public void init() {
        // 推送连接数（Gauge）
        pushConnectionCount = new AtomicInteger(0);
        Gauge.builder("push_connection_active", pushConnectionCount, AtomicInteger::get)
                .description("当前活跃的 SSE 推送连接数")
                .tag("application", "iboot-admin")
                .register(meterRegistry);

        // 推送成功次数（Counter）
        pushEventSentCounter = Counter.builder("push_event_sent_total")
                .description("推送事件发送成功次数")
                .tag("application", "iboot-admin")
                .register(meterRegistry);

        // 推送失败次数（Counter）
        pushEventFailedCounter = Counter.builder("push_event_failed_total")
                .description("推送事件发送失败次数")
                .tag("application", "iboot-admin")
                .register(meterRegistry);
    }

    public void updatePushConnectionCount(int count) {
        pushConnectionCount.set(count);
    }

    public void recordPushEventSent() {
        pushEventSentCounter.increment();
    }

    public void recordPushEventFailed() {
        pushEventFailedCounter.increment();
    }
}
```

### 11.28 Prometheus 查询示例

**查询最近 5 分钟的 QPS：**
```promql
sum(rate(http_server_requests_seconds_count{application="iboot-admin"}[5m]))
```

**查询 P99 延迟：**
```promql
histogram_quantile(0.99, sum(rate(http_server_requests_seconds_bucket[5m])) by (le))
```

**查询登录成功率：**
```promql
rate(user_login_success_total[5m]) / (rate(user_login_success_total[5m]) + rate(user_login_failed_total[5m])) * 100
```

**查询 JVM 内存使用率：**
```promql
jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100
```

### 11.31 故障排查流程

```
1. 收到告警通知
   │
   ▼
2. 查看 Grafana 仪表板，确认告警指标趋势
   │
   ▼
3. 根据告警类型定位问题：
   ├─ 应用宕机 → 检查应用日志、JVM 状态
   ├─ HTTP 错误率高 → 查看错误日志、异常堆栈
   ├─ 延迟过高 → 分析慢查询、GC 日志
   └─ 资源不足 → 检查 CPU、内存、磁盘使用率
   │
   ▼
4. Kibana 搜索相关日志
   │
   ▼
5. 根据日志和指标分析根因
   │
   ▼
6. 采取措施修复问题
   │
   ▼
7. 验证修复效果，关闭告警
```

### 11.32 性能优化建议

**Prometheus 优化：**

| 优化项 | 建议值 | 说明 |
|--------|--------|------|
| 采集间隔 | 15s | 平衡数据精度和存储成本 |
| 数据保留 | 24h 本地 + Thanos 长期存储 | 减少本地存储压力 |
| 抓取超时 | 10s | 避免慢目标影响整体采集 |
| 最大样本数 | 100000 | 防止内存溢出 |

**Elasticsearch 优化：**

| 优化项 | 建议值 | 说明 |
|--------|--------|------|
| JVM 堆内存 | 512MB-1GB | 日志量适中场景 |
| 分片数 | 1-3 个/索引 | 根据数据量调整 |
| 副本数 | 0-1 | 单节点设为 0 |
| 索引保留 | 30 天 | 定期删除旧索引 |

**Grafana 优化：**

| 优化项 | 建议值 | 说明 |
|--------|--------|------|
| 查询超时 | 60s | 避免长查询阻塞 |
| 最大数据点 | 1000 | 限制图表渲染数据量 |
| 刷新间隔 | 30s | 平衡实时性和负载 |

### B. 监控指标命名规范

遵循 Micrometer 命名规范：

```
<领域>_<操作>_<单位>_<统计类型>

示例：
- user_login_success_total      # Counter 类型
- business_process_duration_seconds  # Timer/Histogram 类型
- cache_hit_rate_percent        # Gauge 类型

```

### C. 日志级别使用规范

| 级别 | 使用场景 |
|------|----------|
| ERROR | 系统错误，需要人工介入 |
| WARN | 异常情况，但系统仍可继续运行 |
| INFO | 重要业务流程节点 |
| DEBUG | 开发调试信息 |
| TRACE | 详细追踪信息（生产环境禁用） |
