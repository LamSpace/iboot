## 概述

### 12.1 设计背景

iBoot 系统采用前后端分离架构，在高并发场景下，数据库成为性能瓶颈。为提升系统响应速度和吞吐量，设计了 L1（Caffeine 本地缓存）+ L2（Redis 分布式缓存）的二级缓存架构。

### 12.2 缓存架构优势

| 特性 | L1（Caffeine） | L2（Redis） | 二级缓存组合 |
|------|---------------|-------------|--------------|
| 访问延迟 | 纳秒级（内存访问） | 毫秒级（网络访问） | L1 命中纳秒级 |
| 存储容量 | 受限（JVM 堆内存） | 大容量（独立存储） | 平衡容量与速度 |
| 数据共享 | 单节点 | 跨节点共享 | 兼顾性能与共享 |
| 适用场景 | 热点数据 | 全量缓存数据 | 分层缓存策略 |

### 12.3 技术选型

| 组件 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| L1 本地缓存 | Caffeine | 3.1.8 | GitHub 高性能缓存库，性能优于 Guava 和 Ehcache |
| L2 分布式缓存 | Redis | 7.x | 成熟的分布式缓存，支持持久化和集群 |
| 缓存管理 | Spring Cache | 6.x | 统一的缓存抽象，支持注解驱动 |
| 序列化 | Jackson | 2.15.x | 支持复杂对象和 Java 8 时间类型 |

### 12.4 性能对比

```
┌────────────────────────────────────────────────────────────┐
│                    访问延迟对比（越低越好）                   │
├────────────────────────────────────────────────────────────┤
│  L1 Caffeine:    ████████░░░░░░░░░░░░░░░░  ~100ns          │
│  L2 Redis:       ████████████████████████░░  ~1ms           │
│  Database:       ██████████████████████████████  ~10ms      │
```

**性能提升：**
- L1 命中：比数据库快 **100 倍**
- L2 命中：比数据库快 **10 倍**
- 典型场景：缓存命中率 80%+，平均响应时间降低 **80%**

## 核心组件

### 12.9 TwoLevelCacheManager（缓存管理器）

**职责：** 作为 Spring Cache 的 CacheManager 实现，管理所有二级缓存实例

**核心字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| l1CacheManager | CaffeineCacheManager | L1 缓存管理器 |
| l2CacheManager | RedisCacheManager | L2 缓存管理器 |
| redisTemplate | RedisTemplate<String, Object> | Redis 操作模板 |
| metricsService | BusinessMetricsService | 业务指标服务 |
| cacheMap | Map<String, TwoLevelCache> | 缓存实例映射（ConcurrentHashMap） |

**核心方法：**

```java
public class TwoLevelCacheManager implements CacheManager {

    public static final String CACHE_EVICT_CHANNEL = "cache:evict";

    @Override
    public Cache getCache(String name) {
        return cacheMap.computeIfAbsent(name, cacheName -> {
            Cache l1Cache = l1CacheManager.getCache(cacheName);
            Cache l2Cache = l2CacheManager.getCache(cacheName);

            if (l1Cache == null || l2Cache == null) {
                log.warn("Failed to create TwoLevelCache for '{}'", cacheName);
                return null;
            }

            log.info("Created TwoLevelCache: {}", cacheName);
            TwoLevelCache cache = new TwoLevelCache(
                cacheName, l1Cache, l2Cache, redisTemplate, CACHE_EVICT_CHANNEL
            );
            cache.setMetricsService(metricsService);
            return cache;
        });
    }

    @Override
    public Collection<String> getCacheNames() {
        Set<String> names = new HashSet<>();
        names.addAll(l1CacheManager.getCacheNames());
        names.addAll(l2CacheManager.getCacheNames());
        return names;
    }

    public TwoLevelCache getTwoLevelCache(String name) {
        return cacheMap.get(name);
    }
}
```

### 12.10 TwoLevelCache（二级缓存实现）

**职责：** 实现 Spring Cache 的 Cache 接口，封装 L1+L2 缓存操作

**核心字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| name | String | 缓存名称 |
| l1Cache | Cache | L1 Caffeine 缓存实例 |
| l2Cache | Cache | L2 Redis 缓存实例 |
| redisTemplate | RedisTemplate | Redis 模板（用于 Pub/Sub） |
| evictChannel | String | 失效消息通道 |
| metricsService | BusinessMetricsService | 指标服务 |

**读取方法（get）：**

```java
@Override
public ValueWrapper get(Object key) {
    // 1. 先查 L1 本地缓存
    ValueWrapper wrapper = l1Cache.get(key);
    if (wrapper != null) {
        log.debug("L1 cache hit: {}:{}", name, key);
        if (metricsService != null) {
            metricsService.recordCacheHit();
        }
        return wrapper;
    }

    // 2. L1 未命中，查 L2 Redis 缓存
    wrapper = l2Cache.get(key);
    if (wrapper != null) {
        log.debug("L2 cache hit, backfill L1: {}:{}", name, key);
        if (metricsService != null) {
            metricsService.recordCacheHit();
        }
        // 回填 L1（反向同步）
        l1Cache.put(key, wrapper.get());
        return wrapper;
    }

    log.debug("Cache miss: {}:{}", name, key);
    if (metricsService != null) {
        metricsService.recordCacheMiss();
    }
    return null;
}
```

**写入方法（put）：**

```java
@Override
public void put(Object key, Object value) {
    if (value == null) {
        return;
    }
    log.debug("Put cache: {}:{}", name, key);
    // 同时写入 L1 和 L2
    l1Cache.put(key, value);
    l2Cache.put(key, value);
}
```

**失效方法（evict）：**

```java
@Override
public void evict(Object key) {
    log.debug("Evict cache: {}:{}", name, key);
    // 清除 L1 和 L2
    l1Cache.evict(key);
    l2Cache.evict(key);
    // 发布缓存失效消息，通知其他节点清除 L1
    publishEvictMessage(key.toString());
}

@Override
public void clear() {
    log.debug("Clear cache: {}", name);
    l1Cache.clear();
    l2Cache.clear();
    // 发布全量清除消息
    publishEvictMessage("*");
}
```

**本地失效方法（供监听器调用）：**

```java
/**
 * 仅清除本地 L1 缓存（供 Pub/Sub 监听器调用）
 */
public void evictLocal(Object key) {
    if ("*".equals(key)) {
        l1Cache.clear();
    } else {
        l1Cache.evict(key);
    }
}
```

**发布失效消息：**

```java
private void publishEvictMessage(String key) {
    if (redisTemplate != null && evictChannel != null) {
        String message = name + ":" + key;
        redisTemplate.convertAndSend(evictChannel, message);
        log.debug("Published evict message: {}", message);
    }
}
```

### 12.11 CacheEvictMessageListener（失效消息监听器）

**职责：** 监听 Redis Pub/Sub 消息，清除本地 L1 缓存

**核心方法：**

```java
public class CacheEvictMessageListener implements MessageListener {

    private final TwoLevelCacheManager cacheManager;

    public CacheEvictMessageListener(TwoLevelCacheManager cacheManager) {
        this.cacheManager = cacheManager;
    }

    @Override
    public void onMessage(Message message, byte[] pattern) {
        String messageBody = new String(message.getBody(), StandardCharsets.UTF_8);
        log.debug("Received cache evict message: {}", messageBody);

        try {
            // 消息格式：cacheName:key
            int separatorIndex = messageBody.indexOf(':');
            if (separatorIndex > 0) {
                String cacheName = messageBody.substring(0, separatorIndex);
                String key = messageBody.substring(separatorIndex + 1);

                TwoLevelCache cache = cacheManager.getTwoLevelCache(cacheName);
                if (cache != null) {
                    cache.evictLocal(key);
                    log.debug("Evicted local cache: {}:{}", cacheName, key);
                }
            }
        } catch (Exception e) {
            log.warn("Failed to process cache evict message: {}", messageBody, e);
        }
    }

    /**
     * 注册监听器到 Redis 消息监听容器
     */
    public static void register(RedisMessageListenerContainer container,
                                TwoLevelCacheManager cacheManager) {
        CacheEvictMessageListener listener = new CacheEvictMessageListener(cacheManager);
        MessageListenerAdapter adapter = new MessageListenerAdapter(listener, "onMessage");
        adapter.afterPropertiesSet();
        container.addMessageListener(adapter, new ChannelTopic(TwoLevelCacheManager.CACHE_EVICT_CHANNEL));
        log.info("Registered cache evict message listener on channel: {}",
                TwoLevelCacheManager.CACHE_EVICT_CHANNEL);
    }
}
```

### 12.12 RedisConfig（缓存配置类）

```java
    /**
     * L2 缓存默认过期时间（分钟）
```

**L1 Caffeine 缓存管理器：**

```java
@Bean
public CaffeineCacheManager caffeineCacheManager() {
    CaffeineCacheManager cacheManager = new CaffeineCacheManager();
    cacheManager.setCaffeine(Caffeine.newBuilder()
            .initialCapacity(100)
            .maximumSize(L1_CACHE_MAX_SIZE)
            .expireAfterWrite(Duration.ofMinutes(L1_CACHE_TTL_MINUTES))
            .recordStats());  // 启用统计信息
    return cacheManager;
}
```

**L2 Redis 缓存管理器：**

```java
@Bean
public RedisCacheManager redisCacheManager(RedisConnectionFactory connectionFactory) {
    // 配置 JSON 序列化器，支持类型信息
    ObjectMapper objectMapper = new ObjectMapper();
    objectMapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
    objectMapper.activateDefaultTyping(
            LaissezFaireSubTypeValidator.instance,
            ObjectMapper.DefaultTyping.NON_FINAL,
            JsonTypeInfo.As.PROPERTY);
    objectMapper.registerModule(new JavaTimeModule());

    GenericJackson2JsonRedisSerializer jsonSerializer =
        new GenericJackson2JsonRedisSerializer(objectMapper);

    RedisCacheConfiguration cacheConfig = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(L2_CACHE_TTL_MINUTES))
            .serializeKeysWith(RedisSerializationContext.SerializationPair
                    .fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(RedisSerializationContext.SerializationPair
                    .fromSerializer(jsonSerializer))
            .disableCachingNullValues();  // 不缓存 null 值

    return RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(cacheConfig)
            .build();
}
```

**二级缓存管理器（主管理器）：**

```java
@Bean
@Primary
public CacheManager cacheManager(CaffeineCacheManager caffeineCacheManager,
```

**RedisTemplate 配置（JSON 序列化）：**

```java
@Bean
public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory connectionFactory) {
    RedisTemplate<String, Object> template = new RedisTemplate<>();
    template.setConnectionFactory(connectionFactory);

    Jackson2JsonRedisSerializer<Object> jackson2JsonRedisSerializer =
        new Jackson2JsonRedisSerializer<>(Object.class);

    ObjectMapper objectMapper = new ObjectMapper();
    objectMapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
    objectMapper.activateDefaultTyping(
        LaissezFaireSubTypeValidator.instance,
        ObjectMapper.DefaultTyping.NON_FINAL
    );
    objectMapper.registerModule(new JavaTimeModule());

    jackson2JsonRedisSerializer.setObjectMapper(objectMapper);

    StringRedisSerializer stringRedisSerializer = new StringRedisSerializer();

    // key 采用 String 序列化
    template.setKeySerializer(stringRedisSerializer);
    template.setHashKeySerializer(stringRedisSerializer);
    // value 采用 Jackson JSON 序列化
    template.setValueSerializer(jackson2JsonRedisSerializer);
    template.setHashValueSerializer(jackson2JsonRedisSerializer);

    template.afterPropertiesSet();
    return template;
}
```

### 12.13 缓存配置参数

### 12.14 缓存注解使用

**@Cacheable（查询缓存）：**

```java
@Service
public class ConfigApplicationService {

    public static final String CACHE_NAME = "sys_config";

    /**
     * 根据配置键获取配置值
     * 使用二级缓存：先查本地 Caffeine，再查 Redis，最后查数据库
     */
    @Cacheable(value = CACHE_NAME, key = "#configKey", unless = "#result == null")
    public String getConfigValue(String configKey) {
        return configRepository.findByConfigKey(configKey)
                .map(Config::getConfigValue)
                .orElse(null);
    }
}
```

**参数说明：**

| 参数 | 说明 | 示例 |
|------|------|------|
| value | 缓存名称（对应缓存配置） | `"sys_config"` |
| key | 缓存键（SpEL 表达式） | `"#configKey"`, `"#user.id"` |
| condition | 缓存条件（SpEL 表达式） | `"#result != null"` |
| unless | 排除条件（SpEL 表达式） | `"#result == null"` |
| cacheManager | 指定缓存管理器 | `"cacheManager"` |

**@CacheEvict（清除缓存）：**

```java
@Service
public class ConfigApplicationService {

    /**
     * 创建系统配置
     * 保存成功后清除该配置键的缓存
     */
    @Transactional(rollbackFor = Exception.class)
    @CacheEvict(value = CACHE_NAME, key = "#config.configKey")
    public Config createConfig(Config config) {
        // ...
    }

    /**
     * 更新系统配置
     * 保存成功后清除该配置键的缓存
     */
    @Transactional(rollbackFor = Exception.class)
    @CacheEvict(value = CACHE_NAME, key = "#config.configKey")
    public boolean updateConfig(Config config) {
        // ...
    }

    /**
     * 刷新缓存
     * 清除所有系统配置缓存

    public void refreshCache() {
        log.info("配置缓存已清除，将在下次查询时重新加载");
    }

    /**
     * 清除指定配置键的缓存

    @CacheEvict(value = CACHE_NAME, key = "#configKey")
    public void evictConfigCache(String configKey) {
        log.debug("清除配置缓存：{}", configKey);
```

## 分布式缓存一致性

### 12.17 一致性模型

iBoot 采用 **最终一致性** 模型，通过 Redis Pub/Sub 实现跨节点缓存失效通知：

```
┌─────────────────────────────────────────────────────────────┐
│                    分布式缓存一致性保证                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 同一节点的读写一致性：强一致（先写后读立即生效）            │
│  2. 跨节点的一致性：最终一致（秒级延迟）                       │
│  3. 缓存与数据库一致性：最终一致（事务提交后清除缓存）         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 12.18 失效消息格式

```
消息通道：cache:evict
消息格式：cacheName:key
消息示例：sys_config:user:1001
全量清除：sys_config:*
```

### 12.19 多节点部署场景

```
┌─────────────────────────────────────────────────────────────────┐
│                         Nginx 负载均衡                           │
└─────────────────────────┬───────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
        ▼                 ▼                 ▼
   ┌─────────┐       ┌─────────┐       ┌─────────┐
   │ 节点 A   │       │ 节点 B   │       │ 节点 C   │
   │ L1 Cache│       │ L1 Cache│       │ L1 Cache│
   └────┬────┘       └────┬────┘       └────┬────┘
        │                 │                 │
        └─────────────────┼─────────────────┘
                          │
                          ▼
                   ┌─────────────┐
                   │    Redis    │
                   │  Pub/Sub    │
                   │ cache:evict │
                   └─────────────┘
```

**场景说明：**

1. **节点 A 更新数据**
   - 更新数据库
   - 更新节点 A 的 L1+L2 缓存
   - 发布失效消息到 `cache:evict` 通道

2. **节点 B、C 接收消息**
   - 监听器收到消息
   - 解析 cacheName 和 key
   - 清除本地 L1 缓存

3. **后续请求**
   - 节点 B、C 的 L1 未命中
   - 从 L2 Redis 获取（已是最新数据）
   - 回写到 L1

### 12.20 注意事项

| 场景 | 处理方式 |
|------|----------|
| 单节点更新 | 本地清除 L1+L2，发布 Pub/Sub 消息 |
| 批量更新 | 使用 `allEntries=true` 清除全部 |
| 监听器异常 | 日志记录，不影响主业务流程 |
| 消息丢失 | Redis 重启可能导致消息丢失，极端情况下需要等待 L2 过期 |

## 缓存使用指南

### 12.21 适用场景

**适合缓存的数据：**

- ✅ 频繁读取、较少修改的数据（如系统配置、数据字典）
- ✅ 计算成本高、结果稳定的数据（如统计报表）
- ✅ 热点数据（如首页推荐、热门商品）
- ✅ 权限相关数据（如用户信息、菜单权限）

**不适合缓存的数据：**

- ❌ 频繁变化的数据（如库存数量、秒杀状态）
- ❌ 实时性要求极高的数据（如股票价格、竞拍状态）
- ❌ 大数据量且访问分散的数据（如历史日志）
- ❌ 包含敏感信息且需要实时撤销的数据

### 12.22 最佳实践

**1. 合理设置缓存键**

```java
// ✅ 推荐：使用业务唯一标识作为 key
@Cacheable(value = "sys_user", key = "#userId")
public User getUserById(Long userId) { ... }

// ✅ 推荐：组合键使用 SpEL
@Cacheable(value = "sys_menu", key = "#userId + ':menus'")
public List<Menu> getUserMenus(Long userId) { ... }

// ❌ 避免：使用完整对象作为 key
@Cacheable(value = "sys_user", key = "#user")  // 用户对象可能很大
public User getUser(User user) { ... }
```

**2. 设置合理的过期时间**

```java
// 热数据：1-5 分钟
// 温数据：10-30 分钟
// 冷数据：1 小时或更长

// 数据字典（变化少）
@Cacheable(value = "sys_dict", key = "#type")
public List<DictData> getDictData(String type) {
    // L2 默认 30 分钟，适合字典数据
}
```

**3. 避免缓存穿透**

```java
// ✅ 推荐：unless 排除 null 值
@Cacheable(value = "sys_config", key = "#configKey", unless = "#result == null")
public String getConfigValue(String configKey) { ... }

// ✅ 推荐：返回 Optional 而非 null
public Optional<Config> getConfig(String key) {
    return Optional.ofNullable(configRepository.findByKey(key));
}
```

**4. 事务与缓存的顺序**

```java
// ✅ 推荐：事务提交后清除缓存
@Transactional(rollbackFor = Exception.class)
@CacheEvict(value = CACHE_NAME, key = "#config.configKey")
public Config updateConfig(Config config) {
    // 事务提交后，@CacheEvict 才执行
    return configRepository.save(config);
}

// ❌ 避免：事务未提交就清除缓存
@CacheEvict(...)
public Config updateConfig(Config config) {
    // 如果事务回滚，缓存已清除但数据未更新
    return configRepository.save(config);
}
```

**5. 批量操作缓存处理**

```java
// 场景：批量更新用户状态
@Transactional
@CacheEvict(value = "sys_user", allEntries = true)
public void batchUpdateStatus(List<Long> userIds, Integer status) {
    for (Long id : userIds) {
        userRepository.updateStatus(id, status);
    }
    // 直接清除所有用户缓存，简单可靠
}

// 或使用循环清除单个 key
@Transactional
public void batchUpdateStatus(List<Long> userIds, Integer status) {
    for (Long id : userIds) {
        userRepository.updateStatus(id, status);
        configApplicationService.evictConfigCache("user:" + id);

```

```java
        cacheTotalCount.incrementAndGet();

    /**
     * 记录缓存未命中
     */
    public void recordCacheMiss() {
        cacheTotalCount.incrementAndGet();

     * 计算缓存命中率

        if (total == 0) return 0.0;
        return (double) cacheHitCount.get() / total * 100;
    }
```

## 性能优化

### 12.28 缓存容量规划

**L1 Caffeine 配置优化：**

| 场景 | maximumSize | expireAfterWrite | 说明 |
|------|-------------|------------------|------|
| 高并发热点 | 5000 | 2 分钟 | 限制容量，频繁刷新 |
| 一般业务 | 10000 | 5 分钟 | 默认配置 |
| 读多写少 | 20000 | 10 分钟 | 更大容量，更长过期 |

**内存估算：**

```
L1 内存占用 ≈ 条目数 × 平均每条目大小

假设平均每条目 1KB：
- maximumSize=10000 → 约 10MB 堆内存
- maximumSize=20000 → 约 20MB 堆内存

建议：L1 总内存占用不超过 JVM 堆内存的 5%
```

**L2 Redis 容量规划：**

```
Redis 内存 = 条目数 × 平均每条目大小 × 副本数

假设：
- 平均每条目 2KB（JSON 序列化后）
- 10 万条目
- 无副本（单节点）

Redis 内存 ≈ 100000 × 2KB = 200MB

建议：预留 30% 冗余空间
```

### 12.33 缓存击穿

**问题：** 热点 key 过期瞬间，大量请求击穿缓存到数据库

**解决方案：**

```java
// 1. 互斥锁（分布式锁）
public String getConfigValueWithLock(String configKey) {
    String value = redisTemplate.opsForValue().get("sys_config:" + configKey);
    if (value != null) {
        return value;
    }

    // 获取分布式锁
    RLock lock = redissonClient.getLock("lock:config:" + configKey);
    if (lock.tryLock()) {
        try {
            // 双重检查
            value = redisTemplate.opsForValue().get("sys_config:" + configKey);
            if (value != null) {
                return value;
            }

            // 查询数据库
            value = configRepository.findByConfigKey(configKey)
                    .map(Config::getConfigValue)
                    .orElse(null);

            if (value != null) {
                redisTemplate.opsForValue().set("sys_config:" + configKey, value);
            }
            return value;
        } finally {
            lock.unlock();
        }
    } else {
        // 获取锁失败，等待后重试
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        return getConfigValueWithLock(configKey);
    }
}

// 2. 逻辑过期（永不过期，后台异步刷新）
public class CacheItem<T> {
    private T data;
    private long expireTime;  // 逻辑过期时间

    public boolean isExpired() {
        return System.currentTimeMillis() > expireTime;
    }
}
```

```java
    @Cacheable(value = "local_config", key = "#configKey", cacheManager = "caffeineCacheManager")
    public String getConfigValueFallback(String configKey) {
        // 从本地配置或默认值返回
        return defaultConfigMap.get(configKey);
    }
}
```

### 12.35 数据不一致

**问题：** 缓存与数据库数据不一致

**排查步骤：**

1. 检查事务与缓存清除的顺序
2. 检查 Pub/Sub 消息是否正常传播
3. 检查 L2 过期时间是否合理
4. 检查是否有绕过缓存的直接数据库操作

**解决建议：**

```java
// 1. 确保事务提交后清除缓存
@Transactional(rollbackFor = Exception.class)
@CacheEvict(value = CACHE_NAME, key = "#config.configKey", beforeInvocation = false)
public Config updateConfig(Config config) {
    return configRepository.save(config);
}

// 2. 使用 beforeInvocation=true 的场景区
// 当方法可能抛出异常时，beforeInvocation=false 更安全
@CacheEvict(value = CACHE_NAME, allEntries = true, beforeInvocation = false)
public void batchUpdate() { ... }

// 3. 定期校验缓存与数据库一致性
@Component
public class CacheConsistencyChecker {

    @Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
    public void checkAndFix() {
        // 对比缓存与数据库
        // 修复不一致数据
    }
}
```
