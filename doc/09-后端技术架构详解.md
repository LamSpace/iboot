### 9.2 设计原则

1. **DDD（领域驱动设计）**: 将业务逻辑封装在领域模型中，保持业务与技术的分离
2. **分层架构**: 接口层、应用层、领域层、基础设施层职责清晰
3. **RESTful API**: 遵循 REST 规范，提供统一的 HTTP 接口
4. **安全优先**: 多层安全防护，包括认证、授权、审计
5. **可观测性**: 完善的日志、指标、追踪体系

### 9.10 MapStruct 对象映射

MapStruct 是一个高性能的对象映射框架，在编译时生成类型安全的映射代码。

**核心特性：**
- **编译时代码生成**：运行时无开销，性能优秀
- **类型安全**：编译期检查映射配置
- **零依赖**：生成的代码不依赖 MapStruct 运行时
- **Lombok 集成**：通过 lombok-mapstruct-binding 与 Lombok 无缝协作

**Mapper 接口位置：**
```
iboot-admin/src/main/java/com/iboot/admin/application/mapper/
├── system/
│   ├── UserMapper.java      # 用户映射
│   ├── DeptMapper.java      # 部门映射（支持树形）
│   ├── RoleMapper.java      # 角色映射
│   ├── MenuMapper.java      # 菜单映射（支持树形）
│   ├── PostMapper.java      # 岗位映射
│   ├── DictMapper.java      # 字典映射
│   ├── ConfigMapper.java    # 配置映射
│   └── NoticeMapper.java    # 公告映射
├── job/
│   └── JobMapper.java       # 任务映射
├── message/
│   ├── MessageMapper.java   # 消息映射
│   └── MessageTemplateMapper.java  # 消息模板映射
└── FileMapper.java          # 文件映射
```

**使用示例：**
```java
@Mapper(componentModel = "spring", injectionStrategy = InjectionStrategy.CONSTRUCTOR)
public interface UserMapper {

    // DTO 转 Entity
    @Mapping(target = "id", ignore = true)
    @Mapping(target = "avatar", defaultValue = "")
    User toEntity(CreateUserRequest request);

    // Entity 转 DTO
    @Mapping(target = "roleNames", source = "roles")
    UserResponse toResponse(User user);

    // List 映射（自动遍历）
    List<UserResponse> toResponseList(List<User> users);
}
```

**常用注解：**

| 注解 | 描述 | 示例 |
|------|------|------|
| `@Mapping(source, target)` | 字段映射 | `@Mapping(source = "userName", target = "username")` |
| `@Mapping(ignore = true)` | 忽略字段 | `@Mapping(target = "id", ignore = true)` |
| `@Mapping(defaultValue)` | 默认值 | `@Mapping(target = "status", defaultValue = "1")` |
| `@Mappings` | 多个映射 | `@Mappings({@Mapping(...), @Mapping(...)})` |
| `@Named` | 自定义映射方法 | `@Mapping(qualifiedByName = "formatDate")` |

**注意事项：**
1. 映射接口必须使用 `interface`，不能使用抽象类
2. 字段名相同时可自动映射，无需 `@Mapping` 注解
3. 复杂对象映射建议使用 `source` 属性明确指定源字段
4. 与 Lombok 合用时，确保 `lombok-mapstruct-binding` 依赖已添加
5. 生成的实现代码位于 `target/generated-sources/annotations/` 目录

### 9.12 各层职责说明

#### 接口层（Interfaces）

**职责：**
- 接收 HTTP 请求，进行参数校验和格式转换
- 调用应用服务处理业务
- 返回统一的响应格式
- 处理认证和授权

**示例：**
```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    @Autowired
    private UserService userService;

    @PostMapping
    public Result<UserResponse> createUser(@Valid @RequestBody CreateUserRequest request) {
        UserResponse response = userService.createUser(request);
        return Result.success(response);
    }
}
```

#### 应用层（Application）

**职责：**
- 编排业务流程
- 调用领域服务和仓储
- 处理事务管理
- 发布领域事件

**示例：**
```java
@Service
@Transactional
public class UserService {

    @Autowired
    private UserRepository userRepository;

    @Autowired
    private PasswordEncoder passwordEncoder;

    public UserResponse createUser(CreateUserRequest request) {
        // 检查用户名是否存在
        if (userRepository.existsByUsername(request.getUsername())) {
            throw new BusinessException("用户名已存在");
        }

        // 创建用户实体
        User user = User.builder()
            .username(request.getUsername())
            .password(passwordEncoder.encode(request.getPassword()))
            .nickname(request.getNickname())
            .email(request.getEmail())
            .deptId(request.getDeptId())
            .status(1)
            .build();

        // 保存用户
        userRepository.save(user);

        // 返回响应 DTO
        return convertToResponse(user);
    }
}
```

#### 领域层（Domain）

**职责：**
- 实现核心业务逻辑
- 定义领域模型和规则
- 定义仓储接口

#### 基础设施层（Infrastructure）

**职责：**
- 实现仓储接口
- 数据持久化
- 调用外部服务
- 提供技术支撑

**示例：**
```java
@Repository
public class UserRepositoryImpl implements UserRepository {

    @Autowired
    private UserMapper userMapper;

    @Override
    public User findById(Long id) {
        UserPO po = userMapper.selectById(id);
        return convertToDomain(po);
    }

    @Override
    public void save(User user) {
        UserPO po = convertToPO(user);
        if (user.getId() == null) {
            userMapper.insert(po);
            user.setId(po.getId());
        } else {
            userMapper.update(po);
        }
    }

    @Override
    public boolean existsByUsername(String username) {
        return userMapper.existsByUsername(username) > 0;
    }
}
```

## SSE 推送架构

### 9.25 SSE 推送概述

iBoot 采用 SSE（Server-Sent Events）技术实现服务器向客户端的实时消息推送。SSE 是一种基于 HTTP 的服务器推送协议，使用 `text/event-stream` 格式通过长连接实现服务器到客户端的单向数据传输。

**技术选型理由：**

| 对比项 | SSE | WebSocket |
|--------|-----|-----------|
| 协议 | HTTP 协议 | 独立协议 |
| 通信方向 | 单向（服务器→客户端） | 双向 |
| 连接建立 | 简单 HTTP 请求 | 需要握手 |
| 重连机制 | 原生支持 | 需手动实现 |
| 适用场景 | 消息推送、实时通知 | 实时聊天、在线游戏 |

**功能特性：**
- 实时消息推送：新消息到达时实时推送到客户端
- 消息已读通知：多端同步消息已读状态
- 在线状态感知：实时统计在线用户数量
- 心跳保活：自动心跳机制保持连接活跃
- 断线重连：客户端自动重连机制
- 集群支持：基于 Redis Pub/Sub 实现多实例消息广播

### 9.27 核心组件

| 组件 | 类名 | 职责 |
|------|------|------|
| SSE 端点 | `PushEndpoint` | SSE 连接入口，处理 `/api/push/connect` 和 `/api/push/status` |
| 连接管理器 | `SseEmitterManager` | 管理所有 SSE 连接（连接池、发送、断开） |
| 推送服务 | `PushEventService` | 推送事件服务，支持单播、组播、广播 |
| Redis 广播器 | `RedisPushEventBroadcaster` | Redis 集群事件广播器 |

### 9.28 推送模式

| 模式 | 描述 | 适用场景 | 方法 |
|------|------|----------|------|
| 单播 | 发送给单个用户 | 私信通知 | `sendToUser(userId, event)` |
| 组播 | 发送给多个用户 | 群组通知 | `sendToUsers(userIds, event)` |
| 广播 | 发送给所有在线用户 | 系统公告 | `broadcast(event)` |

### 9.29 推送事件类型

| 事件类型 | 事件来源 | 数据格式 | 说明 |
|---------|---------|---------|------|
| `com.iboot.push.connection.established` | `/api/push/connect` | `{userId, connectedAt, onlineCount}` | 连接建立成功 |
| `com.iboot.push.new.message` | `/api/messages` | `{messageId, title, content, messageType, priority, senderId, sentAt}` | 新消息推送 |
| `com.iboot.push.message.read` | `/api/messages/read` | `{messageId, userId, readAt}` | 消息已读通知 |
| `com.iboot.push.system.notification` | `/api/system` | `{title, content, type}` | 系统通知 |
| `heartbeat` | `/api/push/connect` | `ping` | 心跳包（30 秒间隔） |

### 9.30 推送流程

```
1. 用户 A 发送消息给用户 B
   │
   ▼
2. MessageApplicationService 保存消息
   │
   ▼
3. 发布 MessageSentEvent 事件（@ApplicationEventPublisher）
   │
   ▼
4. MessagePushListener 监听到事件（@TransactionalEventListener）
   │
   ▼
5. PushEventService 构建 PushEvent
   │
   ▼
6. SseEmitterManager 发送 SSE 消息给用户 B
   │
   ▼
7. 用户 B 的浏览器接收并显示通知
```

### 9.31 Redis 集群支持

多实例部署时，通过 Redis Pub/Sub 实现跨实例消息推送：

```yaml
iboot:
  push:
    enabled: true
    # 广播器类型：LOCAL（单机）或 REDIS（集群）
    broadcaster: redis
    # SSE 配置
    sse:
      timeout: 300000        # 连接超时 (5 分钟)
      heartbeat-interval: 30000  # 心跳间隔 (30 秒)
    # Redis 配置
    redis:
      channel: iboot:push:events
```

**集群推送流程：**
1. 用户 A 连接在实例 1，用户 B 连接在实例 2
2. 用户 A 发送消息给用户 B
3. 实例 1 的 `PushEventService` 发现用户 B 不在本实例
4. 通过 Redis Pub/Sub 广播事件到 `iboot:push:events` 频道
5. 实例 2 监听到事件，推送给用户 B

## API 设计规范

### 9.36 细粒度 API 版本控制

iBoot 项目采用 Spring Framework 7.0+ 的细粒度 API 版本控制机制，通过请求头而非 URL 路径来指定 API 版本号。

#### 架构设计

**配置层：**
```java
@Configuration
public class ApiVersionConfig implements WebMvcConfigurer {
    @Override
    public void configureApiVersioning(ApiVersionConfigurer configurer) {
        configurer
            .useVersionResolver(new HeaderApiVersionResolver("Accept-Version"))
            .setVersionRequired(false);
    }
}
```

**核心组件：**
| 组件 | 类名 | 职责 |
|------|------|------|
| 版本解析器 | `HeaderApiVersionResolver` | 从 `Accept-Version` 请求头解析版本号 |
| 版本配置 | `ApiVersionConfig` | 配置版本解析器和版本要求 |
| Controller 方法 | `@GetMapping(version = "1", ...)` | 在映射注解中声明版本号 |

#### 版本控制方式

| 特性 | 说明 |
|------|------|
| 版本请求头 | `Accept-Version: 1` |
| 注解方式 | `@GetMapping(version = "1", value = "/list")` |
| 版本可选 | `setVersionRequired(false)` 允许无版本号请求 |

#### Controller 实现示例

```java
@RestController
@RequestMapping("/api/user")
public class UserController {

    // API v1 版本 - 用户列表查询
    @GetMapping(version = "1", value = "/list")
    public Result<PageResult<UserResponse>> list(...) {
        // API v1 版本实现
    }

    // API v2 版本 - 扩展的数据结构
    @GetMapping(version = "2", value = "/list")
    public Result<PageResult<UserResponseV2>> listV2(...) {
        // API v2 版本实现（返回扩展的数据结构）
    }

    // 无版本号接口 - 适用于所有版本
    @PostMapping
    public Result<UserResponse> create(@Valid @RequestBody CreateUserRequest request) {
        // 不指定版本号，适用于所有版本
    }
}
```

#### 版本兼容性策略

| 策略 | 描述 |
|------|------|
| 向后兼容 | 新版本的接口应尽量保持向后兼容 |
| 默认版本 | 未指定版本号时，使用最新版本接口 |
| 多版本共存 | 旧版本接口应保留至至少 3 个月后方可废弃 |
| 废弃公告 | 废弃接口前应提前公告，给调用方迁移时间 |

#### 与传统 URL 路径版本控制的对比

| 方式 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| 请求头版本控制 | `Accept-Version: 1` + `/api/users` | URL 简洁，便于维护 | 调试不够直观 |
| URL 路径版本控制 | `/api/v1/users` | 直观可见 | URL 冗余，版本升级需改路由 |

iBoot 采用请求头版本控制方式，保持 URL 的简洁性和稳定性。
